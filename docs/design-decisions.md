# 设计决策

> 本文保留产品和技术方案的主要取舍。当前行为以代码、测试及[运行架构](architecture.md)为准。

## 1. 产品边界

归页定位为本地优先的个人社媒账号、本人内容与指标工作台：

- 管理多个本人账号，并用分组、标签、备注和本地别名整理。
- 每个账号使用隔离的内置 Chromium Session，从平台官方入口登录。
- 只读同步本人资料、本人发布内容和本人可见指标。
- 数据统一保存到本地 SQLite，用于内容检索、趋势分析、导出和加密备份。
- 平台通过受限 `platform.adapter` 逐个接入，不宣称“通用抓取”；内置与第三方实现使用同一贡献点合同。

不提供平台写操作、手动 Cookie 或数据导入、DOM/HTML 回退、签名逆向、代理轮换、指纹伪装和无人值守批量账号自动化。

## 2. 为什么使用内置浏览器

直接复用外部 Chrome 会引入配置发现、进程竞争、跨浏览器兼容和会话误用问题。归页改用 Electron 自带 Chromium，并为每个账号固定 `persist:social:<uuid>` Partition：

- 用户始终从平台官方页面登录；登录、扫码和验证由平台页面自己处理。
- 不读取外部浏览器资料，也不把登录状态写进 SQLite 或备份。
- 核验和同步可以复用同一 Session 在后台完成；失效时才把同一 workspace 显示出来。
- 账号浏览器使用独立大窗口，避免在管理界面窄区域内操作完整平台页面。

这是一种会话容器，不是修改 Canvas、WebGL、UA 或网络指纹的“指纹浏览器”。

## 3. 为什么采用 API-only

早期调研以 OpenCLI 为主要参考，也核对了若干平台专项项目。内置平台只移植经过本人账号实测的端点和字段规则；开放插件采用归页自己的 Manifest、权限代理和 QuickJS 沙箱，不直接运行这些项目的 Bridge 或守护进程。

| 项目 | 采用的启发 | 没有采用的部分 |
|---|---|---|
| [jackwener/OpenCLI](https://github.com/jackwener/OpenCLI) | 已登录页面中的同源请求、平台适配器组织、响应捕获 | 原项目的 Bridge、守护进程与执行环境 |
| [ReaJason/xhs](https://github.com/ReaJason/xhs) | 小红书接口与字段交叉核对 | Cookie 输入、签名和公开详情回退 |
| [jackwener/xiaohongshu-cli](https://github.com/jackwener/xiaohongshu-cli) | 公开详情方案的可行性对照 | POST 签名逻辑和 HTML 回退 |
| [Johnserf-Seed/f2](https://github.com/Johnserf-Seed/f2) | 微博、抖音候选接口调研 | 独立 Python 运行时和 Cookie 输入 |
| [d60/twikit](https://github.com/d60/twikit) | X Web GraphQL 操作、游标和响应字段交叉核对 | Cookie 文件、Bearer/CSRF 请求头构造、登录自动化和平台写操作 |

平台 DOM 变化频繁，且页面可见文本难以证明字段身份和分页完整性。固定 JSON 接口可以对白名单、响应大小、ID、计数、分页和前后身份做严格验证，因此成为唯一采集路径。

## 4. 信息架构

应用保留六个一级模块：

| 模块 | 用途 |
|---|---|
| 工作台 | 汇总账号、内容、指标、最近同步和待处理状态 |
| 账号 | 添加、登录、核验、分组、备注和单账号同步 |
| 内容 | 跨账号检索、筛选、备注、标签和打开官方原帖 |
| 数据 | 7/30/90/365 天趋势、账号排行和类型分布 |
| 插件 | 已安装、发现、运行记录、开发插件、授权、配置与计划 |
| 设置 | 主题、导出、清理、备份恢复与软件更新 |

账号详情把“资料与指标”“内容”“设置与备注”放在同一上下文中；完整平台页面在独立账号浏览器窗口打开。同步按钮不要求用户先打开浏览器，登录失效时再给出明确的重新登录入口。

## 5. 账号与内容模型

- 本地备注名可留空；首次绑定成功时采用平台昵称，用户自定义后不再自动覆盖。
- 标签用于轻量检索，分组用于多账号组织；一个账号可属于多个分组。
- “默认账号”当前只是本地标记，尚未驱动工作台或筛选器默认选择。
- 内容保存平台远端 ID、类型、标题、API 摘要、官方原帖 URL、发布时间及本地备注/标签。
- 账号和内容指标保存时间快照；平台没有返回的值保持为空。

## 6. 视觉与交互系统

品牌名为“归页 / Streamfold”，关键词是归档、折叠、流动和本地掌控。当前界面采用中性底色与鸢尾紫主色，支持浅色、深色和跟随系统。

- 正文默认不低于 14px，主要操作、标题和数据使用更高字号与清晰层级。
- 左侧导航可折叠；标题栏、托盘和账号浏览器保持同一主题与窗口状态。
- 弹窗使用应用内统一样式，破坏性操作明确区分并要求确认。
- 桌面图标的矢量母版在 `resources/icons/app-icon.svg`；打包图标位于 `build/`，托盘运行时资源位于 `resources/icons/`。

界面只提供完成操作所需的说明和状态反馈，不把实现免责声明或开发过程提示堆在产品页面中。

## 7. 后续优先级

1. 完成小红书与知乎的更多真实异常场景验收。
2. 在同平台单并发基础上增加串行批量同步和批次进度。
3. 完成 Windows/macOS 代码签名、公证和公开 GitHub Release 更新验收。
4. 按相同安全合同逐个平台评估微博与抖音，而不是一次性开放通用接口。
