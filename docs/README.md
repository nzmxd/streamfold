# 项目文档

文档按使用场景收拢为五份，避免同一事实分散在多处维护。

| 文档 | 适合谁 | 内容 |
|---|---|---|
| [使用指南](user-guide.md) | 使用者 | 添加账号、登录核验、同步、内容与数据、导出、清理、备份和更新 |
| [运行架构](architecture.md) | 开发与审计 | 进程与会话、同步链路、数据库表与迁移、安全边界、更新服务 |
| [平台适配器](platform-adapters.md) | 平台开发者 | 小红书、知乎端点、字段、分页、限制、错误处理与计划中平台 |
| [开发与发布](development.md) | 维护者 | 环境、源码地图、合同变更、测试、CI、安装包、在线更新和当前路线图 |
| [设计决策](design-decisions.md) | 产品与设计 | 内置浏览器、API-only、调研来源、信息架构、账号模型与视觉原则 |

## 阅读顺序

- 第一次使用：根目录 `README.md` → 使用指南。
- 排查同步：使用指南 → 平台适配器 → 运行架构。
- 修改代码：运行架构 → 开发与发布；涉及平台时再读平台适配器。
- 了解方案来源：设计决策。

## 事实优先级

当文档和实现不一致时，按以下顺序判断：

1. 当前共享合同、运行时校验、数据库迁移和测试；
2. 运行架构、平台适配器、开发与发布；
3. 使用指南；
4. 设计决策中的历史取舍。

版本、端点、schema、构件和已实现范围发生变化时，应在同一个提交中更新对应文档。不要为单次问题另建零散说明文件；优先补入以上五份文档的相关章节。
